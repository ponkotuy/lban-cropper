# Photo L版/2L版 アスペクト比変換ツール — 仕様書

## 概要

L版（89×127mm）または2L版（127×178mm）の写真プリント用に、手元の写真のアスペクト比を整えるブラウザ完結アプリ。コンビニプリント等、印刷側に編集UIが無い環境向けに、写真ごとに「クロップ位置を選ぶ」「白帯を入れて元写真をそのまま残す」を1枚ずつ判断できるUIを提供する。

既存の BIRME はバッチ全体のモードしか切り替えられず、Canva はキャンバスベースで重い。Squoosh のような単機能・単一HTMLの軽さを目指す。

## 主要要件

- **単一 HTML ファイルで完結**。ビルドステップなし、フレームワークなし、外部依存なし。Vanilla JS + Canvas API のみ。`file://` 直叩きでも動くこと。
- **完全クライアントサイド**。ファイルは一切ネットワークに出ない。
- **画質劣化を最小化**。詳細は[画質方針](#画質方針)。
- **ピクセル数は固定しない**。L版相当の物理サイズに合わせるための ppi 計算は不要。アスペクト比だけ合わせて、ピクセル数はソース画像から決定する。
- 想定枚数は10〜20枚。1枚ずつ順番に処理するシリアル UI。

## ユーザー体験フロー

1. **起動** — ブラウザで HTML を開くと、用紙サイズ選択（L版／2L版）と大きなドロップエリアが表示される。
2. **写真投入** — 写真を複数ドラッグ＆ドロップ（または「ファイル選択」ボタンから複数選択）。
3. **編集（1枚目）** — 1枚目が画面中央に表示される。デフォルトは写真の向きに合致するクロップモード（例：横長写真なら横クロップ）。クロップ枠が写真の上にオーバーレイ表示される。
4. **モード切替** — ユーザーは3つのモードを切り替えられる：
   - **横クロップ** — L版を横長に使う（1.427:1）。クロップ枠を左右にドラッグして位置決め。
   - **縦クロップ** — L版を縦長に使う（1:1.427）。クロップ枠を上下にドラッグして位置決め。
   - **白帯** — 元写真は無加工。両端または上下に白帯を足してアスペクト比を合わせる。
5. **保存** — 「保存」ボタンで現在の設定通りに書き出し、ファイルが自動ダウンロードされる。即座に次の画像に切り替わる（モードはデフォルトに戻す）。
6. **スキップ** — 「スキップ」ボタンで保存せず次の画像へ。
7. **完了** — 全画像処理後、完了画面（「○枚処理しました」「もう一度」ボタン）。

進捗は常に画面上部に「3 / 20」のように表示する。

## 画面仕様

### 初期画面

- 用紙サイズ選択（ラジオボタンまたはトグル）：「L版 89×127mm」「2L版 127×178mm」。デフォルトは L版。
- ドロップエリア（点線枠の大きな矩形）。エリア外にドロップしても受け付けるとなお良い。
- 「ファイルを選択」ボタン（`<input type="file" accept="image/*" multiple>`）。
- 注記：「画像はブラウザ内でのみ処理されます。アップロードはされません。」

### 編集画面

レイアウト案（縦並び、上から）：

- ヘッダ：進捗（`3 / 20`）、用紙サイズ表示、（編集中は）用紙サイズ変更不可で良い
- モード切替バー：3つのボタン（横クロップ／縦クロップ／白帯）。アクティブなモードがハイライト
- 画像表示エリア：写真が画面に収まるよう縮小表示。クロップモードのときはクロップ枠を半透明オーバーレイで表示。枠外は薄暗くする
- フッタ：「保存して次へ」（プライマリ）、「スキップ」（セカンダリ）、「やり直し」（クロップ位置リセット）

クロップ枠のドラッグ：

- ドラッグ可能な軸は1つだけ（横クロップなら水平、縦クロップなら垂直）。もう一方の軸はソース画像の端まで使い切る。
- マウスとタッチの両対応のため Pointer Events（`pointerdown`/`pointermove`/`pointerup`）を使う。
- 矢印キーで 1px ずつ、Shift+矢印で 10px ずつ移動できると便利（ソース画像の座標系で）。

### 完了画面

- 「○枚を処理しました」
- 「もう一度」ボタン → 初期画面に戻る

## アスペクト比定義

物理サイズに基づく正確な比率を使う。いわゆる「2:3」では微妙にずれる。

| 用紙 | 物理寸法 | 横向きアスペクト比 (W:H) |
|------|---------|--------------------------|
| L版  | 89×127mm | 127:89 ≈ 1.42697 |
| 2L版 | 127×178mm | 178:127 ≈ 1.40157 |

縦向きは上記の逆数。コードでは小数で持って良い：

```js
const RATIOS = {
  L:  { landscape: 127 / 89,  portrait: 89 / 127  },
  '2L': { landscape: 178 / 127, portrait: 127 / 178 },
};
```

## 算出アルゴリズム

ピクセル数は固定しない。**ソース画像の解像度を最大限活かす**方針で出力ピクセル数を決定する。

### クロップモード

ソース画像 `(sw, sh)`、目標アスペクト比 `R = w/h`（横クロップなら 127/89 等）とする。

```
sourceR = sw / sh
if sourceR > R:
    # ソースが横に余っている → 高さは使い切り、幅をクロップ
    cropH = sh
    cropW = round(sh * R)
    # ユーザーは X 方向にドラッグ：0 <= cropX <= sw - cropW
    cropY = 0
else:
    # ソースが縦に余っている → 幅は使い切り、高さをクロップ
    cropW = sw
    cropH = round(sw / R)
    # ユーザーは Y 方向にドラッグ：0 <= cropY <= sh - cropH
    cropX = 0
```

出力 canvas は `cropW × cropH`。`drawImage(img, cropX, cropY, cropW, cropH, 0, 0, cropW, cropH)` で**等倍コピー**（リサンプリングなし）。

横クロップと縦クロップの違いは `R` を `landscape` 比にするか `portrait` 比にするかだけ。

### 白帯モード（フィット）

ソース画像 `(sw, sh)`、目標アスペクト比は **ソース画像の向きに合わせて自動選択**する（横長なら landscape、縦長なら portrait）。これによりソース画像が常に最大限大きくキャンバスに収まる。

```
sourceR = sw / sh
R = sourceR > 1 ? landscape : portrait

if sourceR > R:
    # ソース自体が目標より横長 → 上下に白帯
    outW = sw
    outH = round(sw / R)
    offsetX = 0
    offsetY = floor((outH - sh) / 2)
else:
    # ソース自体が目標より縦長 → 左右に白帯
    outH = sh
    outW = round(sh * R)
    offsetX = floor((outW - sw) / 2)
    offsetY = 0
```

出力手順：

```js
ctx.fillStyle = '#FFFFFF';
ctx.fillRect(0, 0, outW, outH);
ctx.drawImage(img, offsetX, offsetY); // 等倍配置、リサンプリングなし
```

ソース画像の全ピクセルが無加工で出力に含まれる。

### 補足：1ピクセル単位の誤差

`round` の結果アスペクト比が小数で完全一致しないことがあるが、L版印刷で問題になるレベルの差ではない（127/89 は既に無理数に近い比率なので、印刷側で常に微調整されている）。要件として明示済みなのでこのまま進めて良い。

## 画質方針

- **リサンプリングしない**。クロップは `drawImage` で対応する元領域を1:1コピー、白帯はソース画像をそのまま配置。
- **出力フォーマット**：JPEG 固定で良い（コンビニプリント互換性が最優先）。`canvas.toBlob(cb, 'image/jpeg', 0.95)`。0.95 はファイルサイズと画質のバランスが良く、写真用途で目視差はほぼ無い。設定 UI は不要。
- **EXIF 回転対応**：`createImageBitmap(blob, { imageOrientation: 'from-image' })` を使う。これでビットマップ取得時点で回転済み。手動 EXIF パース不要。
- **EXIF メタデータの保持は不要**。`canvas.toBlob` は EXIF を出力しないが、印刷用途なので問題なし。
- **色空間**：sRGB 前提で良い（Canvas のデフォルト）。広色域写真の対応は仕様外。

## 操作系まとめ

| 操作 | 動作 |
|------|------|
| ドラッグ＆ドロップ | 写真を投入 |
| ファイル選択ボタン | 写真を投入 |
| モードボタン（横/縦/白帯） | モード切替。クロップ位置はリセット（中央） |
| 画像エリアをドラッグ | クロップ位置を移動 |
| 矢印キー | クロップ位置を 1px 移動（クロップモード時のみ） |
| Shift+矢印キー | クロップ位置を 10px 移動 |
| Enter | 保存して次へ |
| S | スキップ |
| 1 / 2 / 3 | モード切替（横／縦／白帯） |

## ファイル名規則

`{元のファイル名のベース}_{用紙}_{モード}.jpg`

例：`IMG_1234.JPG` を L版で横クロップした場合 → `IMG_1234_L_crop-h.jpg`

| モード | 接尾辞 |
|--------|--------|
| 横クロップ | `crop-h` |
| 縦クロップ | `crop-v` |
| 白帯 | `fit` |

ブラウザの一括ダウンロード制限に当たる場合は、`<a download>` を順次クリックする間に短い setTimeout（100ms 程度）を挟む。

## 技術メモ

- **単一 HTML**：`<style>` と `<script>` をインラインで含む 1 ファイル。ビルド不要。
- **モジュール構造**（ファイル内）：
  - 状態管理（現在のキュー、インデックス、モード、クロップ位置）
  - 画像読み込み（File → ImageBitmap）
  - 描画（プレビュー）
  - 出力（Canvas → Blob → ダウンロード）
  - イベント処理（ドラッグ、キーボード、ボタン）
- **メモリ管理**：ImageBitmap は明示的に `.close()` で解放。次の画像に進む際に前の ImageBitmap を閉じる。Blob URL も `URL.revokeObjectURL`。
- **プレビューと出力の分離**：プレビューは画面サイズに縮小した canvas で十分。保存時に初めてフル解像度の出力 canvas を作る。50MP 超の写真でも保存時の一瞬だけメモリを使う。
- **OffscreenCanvas 使用可**（出力時のみ）。サポート無い環境のフォールバックは通常の `<canvas>`（`document.createElement('canvas')`、DOM に挿入しない）。
- **HEIC 対応は不要**（Web 標準で読めないため）。`<input accept>` で `image/jpeg,image/png,image/webp` 程度に絞って良い。
- **再ドロップの扱い**：処理中に新たにドロップされたら、確認ダイアログを出してキューを置き換え（または末尾追加するならその旨明示）。シンプル優先で「現在のキューを破棄して新規開始」で良い。

## 受け入れ条件

実装完了の判断基準：

- [ ] HTML 単一ファイルを `file://` で開いて全機能動作する
- [ ] 10枚の写真をドロップ → 1枚ずつ表示される
- [ ] 横クロップ／縦クロップ／白帯の3モードを各写真で個別に選べる
- [ ] クロップ位置をドラッグで動かせる（マウスとタッチの両方）
- [ ] 保存ボタンを押すと現在の写真が JPEG でダウンロードされ、次の写真が表示される
- [ ] ソース画像の縦横比に関係なく、出力のアスペクト比が 127:89（L版横）等と一致している（小数点以下の誤差は許容）
- [ ] クロップモードの出力ピクセル数がソース画像の対応領域と完全に一致する（リサンプリングされていない）
- [ ] 白帯モードの出力にソース画像が等倍で含まれている（リサンプリングされていない）
- [ ] EXIF Orientation が指定された写真が正しい向きで処理される
- [ ] 全画像処理後に完了画面が出る
- [ ] サーバへの通信が一切発生しない（DevTools の Network タブで確認）

## 仕様外（やらないこと）

- 色調・明るさ・コントラスト調整
- 回転（EXIF 自動対応のみ。手動回転は不要）
- 複数画像の並列表示・一括設定
- ZIP まとめダウンロード（ブラウザの個別ダウンロードで十分）
- 設定の `localStorage` 保存
- HEIC/HEIF サポート
- L版・2L版以外の用紙サイズ
- 印刷直接出力（ブラウザ印刷ダイアログは使わない）
- ダークモード・テーマ切替
- 多言語対応（日本語のみで良い）
- アクセシビリティの作り込み（基本的なフォーカス可能性は確保するが ARIA まではやらなくて良い）

## 推奨スタイル

最低限の見た目は欲しい。Squoosh / BIRME のような簡素で目的志向のデザイン。装飾は最小限。ボタンは大きめ、現在のモードがひと目でわかること。日本語フォントは system-ui で良い。
